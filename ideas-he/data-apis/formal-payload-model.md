# יצירת מודל פורמלי לסכימת מטען נתוני ההתראה מנתונים אמיתיים שנלכדו

**קטגוריה:** נתונים, ממשקי API ומערכת אקולוגית למפתחים
**תועד:** 2026-04-06

## הבעיה

הסכימה בפועל של פיד ההתראות של פיקוד העורף מובנת כיום רק באמצעות תצפית של קהילת מגרדי הנתונים ומפתחי האפליקציות. אין מודל קנוני, קריא-מכונה, של מטען הנתונים — אין JSON Schema, אין מסמך OpenAPI, אין מפרט פורמלי — וכתוצאה מכך כל צרכן נאלץ לבצע הנדסה הפוכה לאותם שדות, מקרי קצה ומוזרויות באופן עצמאי.

זהו בזבוז כפול: אותה עבודת מידול משוכפלת בעשרות פרויקטים, ואף פרויקט בודד אינו בטוח לחלוטין שהמודל שלו שלם, מכיוון שאף אחד מהם לא ראה כל מצב שהפיד יכול לייצר.

## פתרון מוצע

יצירת מודל פורמלי למטען הנתונים האמיתי על ידי לכידתו בתנאי אמת וגזירת סכימה קנונית מתעבורה שנצפתה. באופן מעשי:

- להפעיל אספן פסיבי ארוך טווח מול הפיד החי כדי ללכוד כל וריאנט נפרד של מטען הנתונים (סוגי התראות, צורות אזורים, מצבי קצה, תגובות שגיאה).
- לגזור JSON Schema (או שווה ערך) ישירות מהקורפוס שנלכד, כאשר כל שדה מתויג לפי תדירות ודוגמה.
- לפרסם את הסכימה וקורפוס דוגמאות מייצג באופן פתוח כדי שמפתחים אחרים יוכלו לאמת מולו.
- להתייחס לסכימה כמסמך חי, המתחדש כאשר וריאנטים חדשים נצפים בשטח.

זוהי פיסת תשתית קהילתית שימושית ובעלת סיכון נמוך עוד *לפני* שקיים API ציבורי רשמי — ואם יוצע API ציבורי בסופו של דבר (ראו [Provide a public and documented API](public-documented-api.md)), הקורפוס שנלכד והסכימה הנגזרת יהפכו לנקודת התחלה חזקה עבור המפרט הרשמי.

## הערות

- קשור קשר הדוק ל- [Add a "current guidance" field to the alert data feed](../info-during-after/current-guidance-field-in-feed.md): מודל פורמלי של *מהו מטען הנתונים כיום* מקל מאוד לטעון במדויק *מה הוא צריך להכיל גם כן*.
- צריך להיות משולב עם כלי CLI קטן לאימות, כך שכל מפתח יוכל להזרים דרכו מטען נתונים שנלכד ולראות מיד אם הוא תואם.
